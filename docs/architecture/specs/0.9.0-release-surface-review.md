# 0.9.0 release surface review

A critical, end-to-end pass over the user-facing surface (commands, flags,
config keys, env vars, output contracts) ahead of the 0.9.0 release. The goal
is to minimise the breaking changes we will owe users after 0.9.x closes.

Method: enumerated the full citty command tree (41 top-level commands, ~120
subcommands, ~230 flags) directly from `src/cli.ts`, cross-read the Zod config
schema (`src/core/config/schema/*`), the env-var surface, and
`STABILITY.md` / `docs/reference/cli.md`. Claims marked **verified** were
reproduced against the CLI in a sandboxed `HOME`.

Ordering is by cost-of-deferral, not by effort.

---

## P0 — Defects in the surface itself

### 1. `akm search --no-project-context` is a no-op — **verified**

`src/commands/read/search-cli.ts:65` declares an arg literally named
`no-project-context`. citty's parser (mri) applies boolean negation to any
`--no-*` token, so passing `--no-project-context` sets
`args["no-project-context"] = false` — the same value as omitting it.
`getHyphenatedBoolean(args, "no-project-context")` (line 96) therefore always
reads `false`, and the project-context boost is never disabled.

Corroborating: `setup` hits the identical trap and works around it explicitly —
`getHyphenatedBoolean(args, "no-init") || process.argv.includes("--no-init")`
(`src/cli.ts`), with a comment naming the citty behaviour. `search` has no such
workaround. The only test (`tests/integration/commands/search-cli-envelope.test.ts:86`)
asserts the flag does not mutate `process.env` — it never asserts the flag has
its documented effect.

Fix: declare `project-context` as a boolean defaulting to `true` so
`--no-project-context` works through native negation (this also matches the
`--push` / `--no-push` and `--sync` / `--no-sync` pattern already used by
`sync` and `improve`). `AKM_DISABLE_PROJECT_CONTEXT=1` remains the only working
escape hatch until this is fixed.

### 2. `--format md` silently emits JSON everywhere except `health` — **verified**

`--format` is a global flag accepting `json|jsonl|text|yaml|md|html`, but:

- `md` is implemented only by `akm health`. Everywhere else `output()`
  (`src/cli/shared.ts`) falls through to `JSON.stringify` — no error, no
  warning. `akm search foo --format md` prints JSON.
- `html` is implemented only by `akm health`. Everywhere else it *throws*
  `"html output is only available for \`akm health\`"`.

Two unimplemented values, two different failure modes, one of them silent.
Both `md` and `html` are part of the advertised global enum. Either restrict
the enum per command (health-only) or make the unimplemented case a uniform
`UsageError`. Silently returning the wrong format from a documented stable
output contract is the worst of the three options.

### 3. `akm search --type` is unvalidated and its documented enum is stale — **verified**

`akm search foo --type bogustype` returns `{"hits": []}` and exit 0 — a typo is
indistinguishable from a genuine no-match. There is no validation between the
CLI and `db-search`; `src/commands/read/search.ts:114` just defaults to `"any"`.

Separately, the help enum ("skill, command, agent, knowledge, workflow, script,
memory, env, secret, lesson, or any") omits three types that
`PLACEMENT_SPECS` actually ships: `task`, `session`, `fact`. All three are
documented as first-class in `README.md`.

### 4. A blocked-semantic-search warning is emitted into the stable JSON envelope on a default install — **verified**

On a freshly `akm init`'d workspace with no embedding configured, every
`akm search` returns:

```json
"warnings": ["Semantic search is currently blocked. Using keyword search until the semantic backend is healthy again."]
```

`semanticSearchMode` defaults to `auto`, so this is the out-of-box path, not a
degraded one. "Blocked … until healthy again" describes a fault; the actual
state is "not configured". Scripted consumers now have a `warnings` entry on
every single search on a default install.

## P1 — Remove before 0.9.0

0.9.x explicitly permits breaking changes (`STABILITY.md`). Anything on this
list is cheaper to delete now than to carry into 0.10 and delete later.

### 5. `akm improve --auto-accept`

Already deprecated and fully ignored; the help text promises removal "in 0.10".
Carrying a dead, warn-only flag past the release that is *allowed* to break
things, into the series that is *not*, inverts the cost. Delete it in 0.9.0 and
put the migration note in the CHANGELOG.

### 6. `akm config enable|disable <target>`

The only accepted target is `skills.sh` (`src/commands/config-cli.ts:91-130`);
everything else hard-errors. It is a hardcoded alias for
`akm registry add|remove https://skills.sh`. It is currently listed under
**Stable** in `STABILITY.md`, which means we would owe a deprecation cycle for a
one-vendor shim. Remove it, or generalise it to a real component registry.

### 7. `akm graph export --format` collides with the global `--format`

`graph-cli.ts:117` declares a local `--format json|jsonl` meaning "artifact
export format", while the global `--format` (parsed independently from raw argv
by `initOutputMode`) means "envelope format". One token, two consumers, one
value. `akm graph export --format jsonl` silently changes both. Rename to
`--export-format`, or infer from the `--out` extension.

### 8. Five overlapping one-shot migration surfaces, all pinned to 0.9.0

`akm migrate {status,apply}`, `akm backup {create,restore} --for 0.9.0`,
`akm config migrate` ("compatibility alias for the canonical top-level migrate
coordinator"), the second published bin `akm-migrate-storage`, and
`akm help migrate <version>`. `backup` hard-rejects any `--for` value other than
the literal `0.9.0` (`backup-cli.ts:9`).

This is a one-time 0.8→0.9 cutover occupying three top-level commands and a
second binary permanently. Consolidate on `akm migrate`, drop `akm config
migrate`, and mark `backup` + `akm-migrate-storage` as removable at 0.10 in the
CHANGELOG now, so removing them later is a documented plan rather than a break.

### 9. `akm upgrade --skip-checksum`

"Skip checksum verification (not recommended)" — a documented, discoverable way
to disable integrity checking on a self-updating binary. If it exists for
recovery, move it behind an env var (`AKM_UPGRADE_SKIP_CHECKSUM=1`) so it is not
a first-class, tab-completable part of the upgrade contract.

### 10. Split `akm add --allow-insecure`

One flag currently authorises two unrelated risks:

- plain-HTTP source URLs (transport), and
- installing a bundle containing env keys that hijack process execution —
  `LD_PRELOAD`, `PATH` — via `akm env run` (`add-cli.ts:143, 213`).

A user who wants the first is silently granted the second. Split into
`--allow-insecure-transport` and `--allow-dangerous-env-keys` before either
spelling is frozen.

### 11. Test-only env vars are live in the shipped runtime

`AKM_FORCE_INIT_TMP_STASH`, `AKM_FORCE_SETUP_TMP_STASH`,
`AKM_TEST_MIGRATION_CRASH_GAP`, `AKM_TEST_MIGRATION_CRASH_AFTER`,
`AKM_TEST_MIGRATION_FAIL_WORKFLOW_DELETE`, `AKM_EMBED_DETERMINISTIC`,
`AKM_ABLATE_CONTRIBUTORS`, `AKM_DISABLE_SCOPED_UTILITY`, `AKM_NODE_ENTRY`.

Several disable safety checks (`AKM_FORCE_INIT_TMP_STASH` bypasses the
"refusing to persist a tmp stashDir" guard) or deliberately corrupt a
migration. Namespace them consistently (`AKM_TEST_*` / `AKM_INTERNAL_*`) and
state in `STABILITY.md` that they carry no compatibility guarantee — otherwise
the first user who scripts against one makes it a de facto contract.

### 12. Internal tracker IDs leak into `--help`

`"F-6: Bulk-accept all pending proposals…"` (×8 flags on `proposal
accept`/`reject`), `"(F-3 / #384)"` on `feedback --failure-mode`,
`"(F-4 / #385)"` on `history --accept-rate-by-source`, `"(#485)"` on `workflow
start --force`, `"(#22)"` on `workflow status --units`, `"(v1 spec §4.2)"` on
`search --include-proposed`. These mean nothing to users and date the help text.

### 13. `akm init --stashDir` / `--stash-dir` undocumented legacy aliases

`stash-cli.ts:75` reads them from raw argv for "older docs/scripts". They point
at a config key that 0.9.0 **hard-rejects**. Remove with the rest of the
`stashDir` cutover.

## P2 — Rename or re-shape now (breaking, but cheapest today)

### 14. `--filter` vs `--scope` are the same feature under two names

`akm search --filter user=<id> --filter agent=<id> …` and
`akm show --scope user=<id> --scope agent=<id> …` accept identical syntax for
identical multi-tenant scope narrowing. Pick one spelling before either is
frozen.

### 15. `--source` means two different things

On `akm graph *` it is a bundle name or path. On `akm search` / `akm curate` it
is `stash|registry|both`. Rename the latter to `--from` (or the former to
`--bundle`).

### 16. `--target` means five different things

Write destination (`remember`, `import`, `clone`, `improve`, `proposal accept`,
`env|secret *`), bundle to operate on (`tasks *`), run id or workflow ref
(`workflow next|status|report|brief`, positional), source to delete
(`remove`, positional), component to toggle (`config enable`, positional).
Narrow it — `--write-to` / `--bundle` / `<run>` would each be unambiguous.

### 17. `akm bundle` is a half-built noun group — **the largest deferred break**

`src/commands/bundle/bundle-cli.ts:17` states it plainly: "The lifecycle verbs
(`create`/`install`/`update`/`remove`/`sync`/`export`) stay on their existing
top-level commands for 0.9.0". So `akm bundle {list,show,items}` coexists with
top-level `akm {add,list,remove,update,sync}` operating on the same objects,
and `akm list` overlaps `akm bundle list`.

Whatever the end state is, it should be decided and landed in 0.9.0. Moving
five lifecycle verbs under a noun group after the release is the single most
expensive break still on the table.

### 18. `akm list --kind local|managed|remote` predates the bundle/adapter model

The vocabulary does not match `bundles` / adapters anywhere else in 0.9.0.

### 19. Two parallel type vocabularies, one of them the *retired* ref grammar

Plural conceptIds are canonical (`memories/vpn-note`, `skills/code-review`), and
`type:name` is documented as removed with no compatibility parser
(`resolveRef.parseRefInput` rejects it — **verified**:
`akm show knowledge:guide` → `ASSET_NOT_FOUND`).

But the singular form survives in three live surfaces:
- `--type memory` on `search` / `curate`, `--type memories` on `lint`
  (singular *and* plural, on sibling commands);
- **the `akm search "<type>:"` browse grammar** — `parseRefPrefixQuery`
  (`src/indexer/search/fts-query.ts:64`) accepts exactly `memory:` and
  `memory:projectA/` for subtree enumeration. This resurrects the grammar we
  just removed, as query syntax, in the same release.

Align the browse grammar to `memories/` and pick one `--type` spelling now.

### 20. `claude` vs `claude-code`

`VALID_HARNESS_IDS` uses `claude`; the session-log provider and
`akm extract --type` use `claude-code`, reconciled by a canonicalisation shim
(`extract.ts:400-404`). Also: `akm extract --auto` "iterates every available
harness" — that is 2 of the 10 registered harnesses.

### 21. `akm show <ref> toc|section "H"|lines 1 20|frontmatter|full`

A bespoke positional sub-grammar, implemented by rewriting `process.argv` and
injecting undocumented internal flags `--akmView` / `--akmHeading` /
`--akmStart` / `--akmEnd` (`show.ts:596`). The view keywords are reserved words
in ref position. It works, but it is a second parser to maintain forever;
if it stays, the internal flags should at least be rejected when passed
directly.

### 22. Cosmetic: camelCase in help text

`--dryRun` (`migrate apply`, `config migrate`) and `--showSimilar` (`remember`)
are the only camelCase flags among ~230. **Verified** that mri aliases both
spellings, so `--dry-run` and `--show-similar` already work — this is a help-text
fix only, but the help is what users copy into scripts.

### 23. Bare-group defaults are implicit contracts

`bundle`→list, `proposal`→list, `graph`→summary, `config`→list, `env`→list,
`secret`→list … and `tasks`→**doctor**. The odd one out, and it becomes load-
bearing now that `tasks list` / `show` / `remove` were removed in Unreleased.
If `akm tasks list` ever returns, `akm tasks` almost certainly becomes `list`,
which is a break. Worth settling now.

## P3 — Mark experimental / gate

Everything here is *labelled* experimental somewhere but is not gated, and
several are not labelled at all.

### 24. `akm workflow run|brief|report|watch` + YAML v2 programs

Help strings say `EXPERIMENTAL:` and `STABILITY.md` describes the whole surface
as experimental — but nothing gates it, and `akm workflow create <name>.yaml`
puts a user on the experimental path from the stable `create` verb. The
`workflows/program/*` sources still carry live `TODO(R2)` markers for
`max_loops` execution, reducer-schema validation, and retry dispatch. Consider a
config gate (`workflow.experimental: true`) or `AKM_EXPERIMENTAL=1`.

### 25. `akm mv`

`STABILITY.md` and the command description both say Experimental; no gate. It
rewrites refs across every markdown/YAML file in the writable stash and re-keys
index rows. Highest blast radius of anything labelled experimental.

### 26. `akm graph` (8 subcommands) — unclassified

Not mentioned in `STABILITY.md` at all. It exposes SQLite-backed entity/relation
internals (`summary`, `entities`, `relations`, `related`, `entity`, `orphans`,
`export`, `update`). Classify as Experimental.

### 27. `akm health --format html` — unclassified

A bespoke chart template (`src/assets/templates/html`, `AKM_ECHARTS`) plus its
own `md` renderer, both bypassing the normal `output()` path. Classify as
Experimental so the report layout is not a contract.

### 28. The `improve.*` config surface should be declared unstable wholesale

`schema/improve-processes.ts` alone defines ~60 tuning fields across 9
processes — `antiCollapse.minSpecificityRetention`, `cls.adjacentCount`,
`lowValueFilter`, `extract.triage.minScore`, `salience.replayBudget`,
`collapseDetector.recallDropThreshold`, `p90ChunkSecondsDefault`, … Most carry
comments of the form "Default OFF", "WS-3b step 10", "#639", "advisory in v1",
"promotion path: …".

`docs/reference/roadmap.md` says 0.9.x is still "tuning the improve processes …
and settling the final design of those features". A config surface still being
designed should not be inheriting the top-level config's stability posture by
default. `STABILITY.md` currently classifies `akm improve` the *command* as
Evolving and says nothing about `improve.*` config keys. Add an explicit
statement that `improve.strategies.*.processes.*` is experimental and may drop
keys in any 0.9.x/0.10.x release.

Related, same section: `improve.strategies.*.symmetricValence` is already
deprecated in code (`preparation.ts:1833`) but still accepted by the schema.

### 29. `index.*` is not frozen either

`IndexConfigSchema` is a `catchall` over arbitrary pass names;
`index.stalenessDetection` is documented in-code as **RETIRED** but kept
"TOLERATED" so old configs validate; `graphExtractionIncludeTypes` is
"accept-any until Chunk 2 … no hardcoded type allowlist". Either finish the
narrowing in 0.9.0 or declare `index.*` experimental.

### 30. Three unrelated polling/streaming surfaces

`akm log tail --interval-ms --max-duration-ms --max-events`,
`akm workflow watch --stream --interval-ms`,
`akm extract --watch --debounce-ms`. Three long-lived foreground modes with
three flag vocabularies. Unify the knob names before they are scripted against.

## P4 — Defaults worth revisiting

### 31. Interactive `akm setup` installs OS scheduler entries; `akm setup --yes` does not

`stepDefaultImproveTasks` (`src/setup/steps/tasks.ts:38`) registers
`DEFAULT_IMPROVE_TASKS` into cron / launchd / schtasks. The default set includes
`akm improve --strategy frequent` **hourly** and `--strategy consolidate` every
4h — both LLM-calling. The wizard asks "is this a server install?" (which only
toggles the nightly sweep); it never asks whether to install scheduled jobs at
all. `runSetupWithDefaults` (`--yes`) skips the step entirely.

Two problems: recurring third-party API spend enabled by an unrelated question,
and interactive/non-interactive setup producing materially different installs.

### 32. `akm improve` commits **and pushes** by default

`improve.ts:865`: `const push = options.sync?.push ?? improveProfile.sync?.push ?? true`.
Combined with #31, the default install has an hourly cron job that can push to a
git remote unattended. `akm init` runs `git init` on the stash, so git-backed is
also the default. Pushing is a reasonable opt-in; it is a strong default.

### 33. `output` config cannot express the CLI's own format set

`OutputConfigSchema` allows `format: json|yaml|text` only — a user cannot
persist `jsonl`, and (given #2) `md`/`html` are moot. There is also no
`output.shape` even though `--shape` is a first-class axis. Either extend the
config enum to match `OUTPUT_FORMATS` or document the asymmetry.

### 34. Minor default inconsistencies

- `akm hints --detail` defaults to `normal` while the global `--detail`
  defaults to `brief`, and for `hints` `normal` ≡ `full` (only `brief`
  differs) — a global flag shadowed with different semantics and a different
  default.
- `akm curate --limit` defaults to the magic string `"4"`.
- Every numeric flag is declared `type: "string"` and parsed downstream, so
  `--limit abc` fails at parse time rather than as a usage error.

## P5 — `STABILITY.md` gaps (docs-only, but do them before release)

The whole point of `STABILITY.md` is that users can tell what is safe to script.
Right now twelve top-level commands are simply absent from it.

### 35. Unclassified top-level commands

`health`, `graph`, `bundle`, `upgrade`, `registry`, `migrate`, `backup`,
`lint`, `extract`, `hints`, `completions`, `init` — none appear in the Stable /
Evolving / Experimental lists (`health` appears once, in the exit-code table).
Also unclassified: every `env` / `secret` **write** verb (`create`, `set`,
`unset`, `remove`, `run`, `path`, `export`) — only `env list` / `secret list`
are covered.

### 36. Stale enumerations in shipped text

| Where | Says | Actual |
| --- | --- | --- |
| `STABILITY.md` agent backends | claude, opencode, codex, gemini, aider | + `copilot`, `pi`, `amazonq`, `openhands`, `opencode-sdk` (10 total) |
| `improve --strategy` help | default, quick, thorough, memory-focus, graph-refresh | + `frequent`, `consolidate`, `catchup`, `reflect-distill`, `proactive-maintenance` (10 total) |
| `search --type` help | 10 types | + `task`, `session`, `fact` (13) |
| `init --set-default` help | "write stashDir to config.json" | writes `bundles` / `defaultBundle`; `stashDir` is hard-rejected |
| `setup --dir` help | "overrides stashDir in config" | same |
| `config get` help | "for example: embedding, stashDir" | same |
| `show` legacy-flag error | "`akm show knowledge:guide toc`" | `type:name` is removed |
| `mv`, `remember --xref/--supersedes`, `proposal show/diff/accept`, `log --ref`, `clone` help | `memory:projectA/old-note`, `skill:akm-dream`, `knowledge:auth-flow`, `type:name`, `npm:@scope/pkg//script:deploy.sh` | all retired `type:name` spellings |

### 37. Env vars have no stability statement

`AKM_DISABLE_PROJECT_CONTEXT` is named in `akm search --help` — so it is a de
facto public interface — but appears in no stability doc. `AKM_EVENT_SOURCE` is
injected into every agent/task child process. `AKM_STASH_DIR`, `AKM_DATA_DIR`,
`AKM_STATE_DIR`, `AKM_CONFIG_DIR`, `AKM_CACHE_DIR`, `AKM_LLM_API_KEY`,
`AKM_EMBED_API_KEY`, `AKM_ENGINE_<NAME>_API_KEY`, `AKM_REGISTRY_URL`,
`AKM_NPM_REGISTRY`, `AKM_SQLITE_JOURNAL_MODE`, `AKM_VERBOSE`, `AKM_DEBUG`,
`AKM_NON_INTERACTIVE` are all load-bearing and undocumented as contracts. Add an
env-var table to `STABILITY.md` split into supported vs internal (see #11).

## Suggested cut line for 0.9.0

If only part of this lands, the ordering that minimises future breaks is:

1. **P0 #1–#4** — these are defects, not design debt.
2. **#17 (`akm bundle` lifecycle verbs)** — decide it now; it is the one break
   that gets dramatically more expensive after the release.
3. **#19 (the `<type>:` browse grammar)** — shipping the grammar we just
   removed, as query syntax, in the same release, guarantees a second cutover.
4. **P1 removals #5–#13** — all cheap, all one-way doors after 0.9.x.
5. **#28 / #29** — a one-paragraph `STABILITY.md` change that buys freedom to
   keep tuning `improve.*` and `index.*` through 0.10 without owing migrations.
6. **P5 #35–#37** — pure documentation, and the difference between a stability
   policy that works and one that has holes in it.
