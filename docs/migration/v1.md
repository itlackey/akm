# Migrating to akm v1

> **Status note (2026-04-27):** This guide is under active rewrite. Much of the content below still reflects the earlier `v0.6.0` architecture-refactor baseline and does **not** yet describe the planned v1 reflection/proposal surfaces. Treat it as historical migration context, not the authoritative pending-work plan. For current sequencing, use `docs/reviews/v1-implementation-plan.md` and `docs/reviews/v1-agent-reflection-issues.md`.

## Historical baseline

The v1 architecture spec landed during the 0.6.0 release cycle. Most users
upgrading from a 0.5.x or earlier 0.6.0 pre-release have nothing to do — the
config loader auto-migrates the legacy keys in memory and persists the new
shape on the next `akm config` write.

The two cases that **do** require manual action are:

1. You had an `openviking`-typed source. Remove it.
2. You had `writable: true` on a `website` or `npm` source. Drop the flag or
   re-add the path as a `filesystem` source.

If neither applies, you can stop reading. Everything else in this guide is
informational.

---

## TL;DR — locked decisions for v1

These four decisions are now part of the v1 contract and shape the behaviour
of every command that touches sources:

1. **`writable` defaults to `true` on `filesystem` sources, `false`
   otherwise.** You own the directories you point akm at. Read-only
   filesystem sources need an explicit `writable: false`.
2. **Registry results are off by default in `akm search`.** Default search
   output is source-only and stays scannable. Pass `--include-registry`
   (or `--source registry|both`) to merge registry hits.
3. **`defaultWriteTarget` is an explicit root config key** that names which
   source receives writes. Resolution order:
   `--target` flag → `defaultWriteTarget` → `stashDir` (the working stash
   created by `akm init`). If none resolve, akm throws
   `ConfigError("no writable source configured; run \`akm init\`")`.
4. **`writable: true` on `website` and `npm` is rejected at config load.**
   The next `sync()` would clobber writes. The loader throws `ConfigError`
   naming the source and kind, with hint:
   *"writable: true is only supported on filesystem and git sources."*

## OpenViking source removed (Phase 1)

akm v1 removes the `openviking` source provider. The provider required a
live API server for every search and show operation, which prevented the
unified FTS5 indexing pipeline that v1 builds on. API-backed sources will
return in a future `QuerySource` tier, designed as a separate query layer
that competes fairly with local FTS5 results.

### Error you will see

A config that contains an `openviking`-typed source fails to load:

```
openviking is not supported in akm v1. API-backed sources will return as a
separate QuerySource tier post-v1. Remove the source from your config (akm
config sources remove <name>) or downgrade to 0.6.x. See docs/migration/v1.md.
```

The error class also carries an actionable hint:

```
Run `akm config sources remove <name>` then re-run.
```

### Remediation

```sh
akm config list                       # find the OpenViking source name
akm config sources remove <name>      # remove it
```

If you have multiple OpenViking sources, repeat for each one. After they're
gone, every command works again.

### Downgrade path

If you depend on OpenViking and cannot wait for the post-v1 `QuerySource`
tier, pin to 0.6.x's last `openviking`-supporting release or to 0.5.x:

```sh
npm install -g akm-cli@0.5
```

Your stash content and config (minus `openviking` entries) are compatible
with 0.5.x.

## `stashes[]` → `sources[]` (Phase 2)

The config key `stashes[]` was renamed to `sources[]`. The internal type
`StashConfigEntry` became `SourceConfigEntry`; the directory
`src/stash-providers/` became `src/sources/providers/`.

**No user action is required.** The loader does the right thing:

- A config with only `stashes[]` loads cleanly. The loader rewrites it
  in memory to `sources[]` and emits one deprecation warning to stderr:
  *Config key `"stashes"` is deprecated; rename it to `"sources"` in your
  config file (`akm config edit`).*
- A config with both keys throws `ConfigError("config has both 'sources'
  and 'stashes' keys; remove 'stashes' to silence the legacy migration
  path")`.
- The renamed key is persisted on the next `akm config` write
  (`akm config set …`, `akm add …`, etc.).

New configs should write `sources[]` directly. The legacy key is read for
one release cycle only; it will be removed in a future major.

The CLI surface is unchanged: `akm config sources …`, `akm list`, and
`akm add` all behave the same as before the rename.

## `SourceProvider` interface simplified (Phase 3)

The `LiveStashProvider` / `SyncableStashProvider` split is gone. Every
provider now implements a single minimal interface:

```ts
interface SourceProvider {
  name: string;
  kind: "filesystem" | "git" | "website" | "npm";
  init(): Promise<void>;
  path(): string;        // absolute path on disk
  sync?(): Promise<void>; // optional; called by `akm update`
}
```

This is internal — no user-visible config or CLI impact. Plugin authors
should re-read [`docs/technical/v1-architecture-spec.md`](../technical/v1-architecture-spec.md)
§2 if they ship a custom provider.

---

## Search and show pipeline restructured (Phase 4)

Search and show no longer fan out across providers. Both consult the
unified FTS5 index, then read files from disk via `provider.path()`.

User impact: none expected. Search ranking is unchanged. Show output is
unchanged. You may notice a small latency improvement on first-run after
`akm index`.

If you scripted around the old `--source <kind>` filter, the same flag is
still accepted; the implementation is just simpler now.

---

## New `writable` flag and `defaultWriteTarget` (Phase 5)

Two new optional fields control where write commands (`akm remember`,
`akm import`, `akm save`, `akm clone`) deposit their output.

### Per-source `writable`

```jsonc
{
  "sources": [
    { "name": "team-stash", "type": "git",        "url": "…",            "writable": true  },
    { "name": "vendor-docs", "type": "website",    "url": "…"                                },
    { "name": "playbooks",   "type": "filesystem", "path": "~/play",      "writable": false }
  ]
}
```

Defaults:

| Source kind  | Default `writable` |
| ------------ | ------------------ |
| `filesystem` | `true`             |
| `git`        | `false`            |
| `website`    | `false`            |
| `npm`        | `false`            |

`writable: true` on `website` or `npm` is **rejected at config load** with
`ConfigError("writable: true is only supported on filesystem and git
sources")`. If you need to author into a checked-out npm package, re-add
the same path as a `filesystem` source.

### Root `defaultWriteTarget`

```jsonc
{
  "defaultWriteTarget": "team-stash",
  "sources": [ … ]
}
```

`defaultWriteTarget` names the source that receives writes when no
`--target` flag is provided.

**Write-target resolution order:**

1. `--target <name>` flag on the command.
2. `defaultWriteTarget` from config.
3. `stashDir` (the working stash created by `akm init`).
4. If none resolve: `ConfigError("no writable source configured; run \`akm init\`")`.

There is no "first writable in `sources[]` order" fallback — that produced
ambiguity without payoff and is intentionally absent.

## Registry providers and Context Hub (Phase 6)

Context Hub is no longer a special-cased provider type. It is just a git
repo. The legacy `type: "context-hub"` value normalizes to `type: "git"`
at load time, so existing configs continue to work; the normalized form is
what `akm config list` shows.

To add Context Hub on a fresh install:

```sh
akm add github:andrewyng/context-hub
```

Or include it as a kit entry in your registry index. The official registry
ships it as a recommended kit.

`akm enable context-hub` and `akm disable context-hub` are removed. Use
`akm config sources remove context-hub` (or edit the entry's `enabled`
flag in your config) to disable it without removing the cached content.

---

## Error hints surface without `--verbose` (Phase 7)

Error classes now own their `hint()` text. The regex-on-message hint chain
in `cli.ts` is gone. Every error printed to stderr now includes the hint
inline:

```
Error: openviking is not supported in akm v1. …
Hint:  Run `akm config sources remove <name>` then re-run.
```

You no longer need `--verbose` to see remediation hints. Scripts parsing
JSON output should rely on the machine-readable `code` and `hint` fields
already present in the error envelope; the textual format is
informational.

---

## DB schema bump (DB_VERSION 8 → 9)

v0.6.0 adds a `workflow_documents` table to `index.db`. On first run after
upgrade, `ensureSchema()` detects the version mismatch and drops + recreates
all `index.db` tables; `usage_events` is preserved via a typed backup. The
next `akm index` rebuilds the index. `workflow.db` (run state) is untouched.

No manual action is required. `akm upgrade` now invokes the new binary's
`akm index` automatically after a successful upgrade so the migration +
rebuild happen up front; pass `--skip-post-upgrade` to defer that step
to the next manual invocation.

## Internal cleanup (no user impact)

These changes affect contributors only and are listed for completeness.

- **Removed re-export shims:** `src/llm.ts`, `src/registry-provider.ts`,
  and `src/ripgrep.ts` are deleted. akm has no public API (CLI-only
  package, no barrel exports), so external consumers should be unaffected.
  Anyone importing from akm internals: switch to the moved paths under
  `src/llm/`, `src/registry/`, and the integrations directory.
- **`src/` reorganized into subdirectories.** Top-level files migrated
  into purpose-named subdirs. The current layout:

  ```
  src/
    cli.ts
    version.ts
    globals.d.ts
    commands/        # one file per CLI verb
    core/            # config, types, errors, refs, write-source
    indexer/         # FTS5 index + lookup
    integrations/
    llm/
    output/          # output shapes + text rendering
    registry/        # registry providers + factory
    setup/           # setup steps, one per file
    sources/         # source providers + factory
    templates/
    wiki/            # wiki-crud, wiki-index, wiki-lint, wiki-ingest
    workflows/
  ```

  Tests and the CLI binary are unchanged. There is no public API, so this
  is purely an internal refactor.

## Verifying the upgrade

Run:

```sh
akm info
```

Expect:

- `version` reports `0.6.0` or higher.
- `sourceProviders` lists only `filesystem`, `git`, `website`, `npm`.
- No `openviking` provider entry.

Then:

```sh
akm config list           # sources[] populated; no warnings
akm list                  # all sources resolve
akm search "<query>"      # indexed content still matches
```

If `akm config list` warns about the legacy `stashes` key, run
`akm config edit` and rename `stashes` to `sources` to silence the warning.

## Rolling back

The auto-migrations are non-destructive. Your `config.json` is rewritten
only when you run a write command (`akm config set`, `akm add`, etc.). If
you need to roll back to 0.5.x:

```sh
npm install -g akm-cli@0.5
```

Restore your `config.json` from a backup if it has been rewritten with
`sources[]` (0.5.x reads `stashes[]` only). Your stash content itself is
unchanged across versions; only the metadata layer evolved.

If you find a regression, please file an issue at
<https://github.com/itlackey/akm/issues> with the output of `akm info` and
a redacted copy of your `config.json`.
