## Work streams

Five issues → **3 parallel streams** (A owns the shared `consolidate.ts`; B and C are isolated). Streams can run concurrently; the sequencing inside A is the only ordering rule.

### WS-A — Memory-dedup & consolidation coverage (`#617` → `#581`)
Single owner (both edit `consolidate.ts`; serialize to avoid conflicts).

**#617 — deterministic near-dup dedup (do first; self-contained)**
- Touchpoints: new `src/commands/improve/dedup.ts`; hook as a pre-pass in `consolidate.ts` before `clusterMemoriesBySimilarity` (consolidate.ts:500) / chunking. Reuse `embedBatch`/`cosineSimilarity` (consolidate.ts:51) + a normalized content-hash.
- Approach: (1) collapse `*.derived.md` + origin near-identical pairs deterministically (hash/strict-cosine ≥ ~0.97, no LLM); (2) collapse content twins above threshold; emit existing consolidate merge/delete ops with provenance preserved. Distinct-but-related items fall through to the normal LLM pool.
- Acceptance: a `.derived` near-identical to its origin collapses with no LLM call; two content twins merge to one; distinct memories never merged; default-off / strict-threshold reproduces current behavior.
- Tests: synthetic near-dup + distinct pairs (deterministic, no network).
- Risk: low — it's a conservative fast path in front of existing machinery.

**#581 — judged-state cache (do after #617)**
- Touchpoints: `consolidate.ts:1115` (`if (opts.incrementalSince)` time-window narrowing + `neighborsPerChanged`), pool selection + `limit` rotation (consolidate.ts:1136); new judged-state store (state.db table: per-memory `{entryKey, contentHash, judgedAt, outcome}`).
- Approach: replace `incrementalSince` time-window narrowing with "judge unless changed" — skip a memory if its content hash matches the last judged hash; this lets a run cover the **full corpus** cheaply (only un-judged/changed memories hit the LLM) instead of a rotating slice. Keep `limit` as a safety cap.
- Acceptance: a second consolidation run over an unchanged corpus makes ~0 LLM calls; a changed/new memory is re-judged; full-corpus coverage no longer requires many rotation cycles.
- Tests: judged-state hit/miss; changed-hash re-judge; migration of the state.db table (characterization snapshot — see cross-cutting).
- Risk: medium — schema migration + pool-selection rewrite; gate on the migration characterization test.

### WS-B — Hot-probation intake buffer (`#604`)
Isolated from WS-A (touches the memory **write/promotion** path + belief-state, not the consolidate pool).
- Touchpoints: memory write path (`promoteProposal` in proposals.ts; extract/distill/remember writers), frontmatter belief-state (reuse `memoryCleanup.beliefStateTransitions` machinery), improve admission of probation memories for review/merge.
- Approach: new memories enter a `probation` belief-state with retrieval/feedback/age counters; promotion to long-term requires crossing a threshold; probation memories are admitted to improve (and to #617/#581 dedup) so dups are caught at intake before they become permanent.
- Acceptance: a newly-written memory starts in probation; promotion only after the criteria; probation never blocks retrieval; cleanly disable-able (default-off → current behavior).
- Tests: probation→promotion transition; criteria gating; default-off parity.
- Risk: medium — touches the write path and belief-state; keep default-off and behind a flag.

### WS-C — Learning-signal tuning (`#614` + `#612`)
Two near-isolated issues; can be one owner sequentially or two sub-streams.

**#614 — symmetric valence weighting (small, do anytime)**
- Touchpoints: proactive-maintenance priority formula (`proactive-maintenance.ts:186` — currently `importance × log(1+retrievalFreq) × recencyDecay / log10(size)`, **no valence term**); the feedback-eligibility sort in `improve.ts` (`getUtilityScoresByIds`); `applyFeedbackToUtilityScore` (MemRL EMA, #386).
- Approach: replace negative-only weighting with a symmetric `|valence|` magnitude term so strong positive and strong negative feedback both drive attention; keep utility as the dominant factor; route high-negative → fix lane, high-positive → reinforce lane. Deterministic.
- Acceptance: equal-magnitude high-positive and high-negative assets rank comparably; utility remains primary; sort is deterministic + unit-tested.
- Risk: low — isolated comparator/term change.

**#612 — auto-accept calibration (do anytime; substrate shipped)**
- Touchpoints: gate (`improve-auto-accept.ts:129` effectiveThreshold, `recordGateDecision`), gate-decision store (proposals.ts, `#577`), new calibration reader (join predicted `confidence` → `auto-accepted`/`auto-rejected` outcome over a rolling window), `health.ts` (calibration summary, mirror the `llmUsage` aggregate + report render), bounded threshold auto-tune in the gate-config resolver.
- Approach: compute calibration (predicted-confidence vs realized accept/reject) from the already-recorded gate decisions + today's `failedByReason`; surface a calibration block in `akm health`; auto-tune `globalThreshold` within a bounded range, logged. **Opt-in.**
- Acceptance: `akm health` shows calibration summaries; threshold tuning is bounded + logged; opt-in; a synthetic gate-decision event stream proves the expected adjustment.
- **Scope cut:** the "feed surprise/prediction error into replay prioritization" bullet depends on replay (#610, 0.10) → land calibration + tuning now, defer the replay hook.
- Risk: low — reads shipped data; auto-tune is bounded + opt-in.

## Phasing (value/risk-ordered)

- **Phase 1 (quick wins, low risk, directly attack the symptom):** `#617` (dedup fast path) + `#614` (valence sort). Both small, self-contained, ship fast. #617 directly reduces the duplicate memories the maintainer is seeing.
- **Phase 2 (coverage + calibration):** `#581` (full-corpus consolidation) + `#612` (calibration). Bigger; build on Phase 1 / shipped substrate. #581 is the engine that clears the backlog #617 can't.
- **Phase 3 (intake prevention):** `#604` (probation buffer) — largest behavioral change; can start in parallel (WS-B) but land last so dedup/coverage are proven first.

Parallelization: WS-A, WS-B, WS-C run concurrently. Within WS-A, #617 → #581. Phase labels order *landing*, not start.
