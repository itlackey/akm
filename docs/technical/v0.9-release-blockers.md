# AKM 0.9 Release Blockers

This ledger is the release authority for the 0.8-to-0.9 task upgrade and scheduler regressions. A blocker is closed only after its regression test, focused suite, and independent review pass. Tests must not modify the developer's live crontab, LaunchAgents, Task Scheduler library, task definitions, or configuration.

## Workflow

1. Reproduce each blocker with a failing regression test or immutable prior-session evidence.
2. Implement the smallest cross-platform fix without a compatibility wrapper or scheduler redesign.
3. Run focused tests on the changed contract.
4. Have an independent reviewer inspect behavior, portability, failure handling, and test strength.
5. Resolve review findings, then run formatting, typecheck, build, all tests, and `tests/release-check.sh`.
6. Run native scheduler acceptance only in disposable CI jobs with unconditional cleanup.

## Migration Code Blockers

| ID | Status | Blocker | Required acceptance |
| --- | --- | --- | --- |
| B01 | Closed locally | Versionless 0.8 tasks, `profile`, and `akm improve --profile` are rejected by the 0.9 reader. | Published 0.8 creates unchanged prompt, workflow, command, and disabled tasks; installed 0.9 migrates, syncs, and runs them without rewriting source files. |
| B02 | Closed locally | A scheduled command that invokes bare `akm` performs a second lookup under the scheduler's restricted `PATH`. | A task created by 0.8 runs under a stripped `PATH` through Bun, Node, and a standalone executable. |
| B03 | Closed locally | Node scheduler registration targets `dist/cli.js` instead of the supported `dist/cli-node.mjs` wrapper. | Packed Node installation reports and executes `cli-node.mjs` through `tasks doctor`. |
| B04 | Closed locally | Interrupted v1 improve rows without `memorySummary` crash long-window health. | Only the known interrupted v1 shape is normalized; complete snapshot selection and denominator accounting remain accurate. |
| B05 | Closed locally | Existing schedules such as `2-22/4` cannot be reconciled, Windows rejects shipped hourly defaults, and launchd/Task Scheduler interval rendering changes cron wall-clock semantics. | Every shipped and persisted schedule translates on each supported backend with cron-equivalent boundaries, or is rejected before installation with an explicit portable error. |
| B06 | Closed locally | Standalone release executables register themselves as a runtime plus a nonexistent script entry. | A compiled binary outside `PATH` reports a one-element invocation and runs a nested `akm --version` task. |
| B07 | Closed locally | Windows command tasks fall back to `/tmp`, and the npm bin launcher assumes `/bin/sh`. | With `HOME` unset, a packed npm install and compiled executable can add, run, inspect, and remove a task on native Windows. |
| B08 | Closed locally | launchd enable overrides survive reinstall/removal; launchd and Task Scheduler syncs are not idempotent. | Disable, remove, reinstall-enabled, and no-op sync tests prove correct state and no unnecessary process interruption. |
| B09 | Closed locally | Scheduler entries do not preserve the resolved non-secret stash/config/data/cache/state context. | All backends execute against the same resolved AKM directories used at registration without serializing credentials or arbitrary environment variables. |
| B10 | Closed locally | The shipped disabled catch-up task is documented as manually runnable, but `tasks run` refuses every disabled task. | Manual invocation runs a disabled task; backend-generated scheduled invocation still skips it. |
| B11 | Closed locally | Invalid sync input can leave an old entry active, forced replacement can lose the old definition, and remove can delete YAML after backend failure. | Failure-injection tests prove stale entries are disabled and definitions/scheduler state are preserved transactionally. |
| B12 | Closed locally | Filesystem task IDs bypass validation, valid IDs can invalidate schtasks XML comments, and cron values permit `%`/newline control syntax. | Shared ID validation and backend escaping/rejection tests cover manually created files and hostile-but-valid paths. |
| B13 | Closed locally | Pre-parse task failures are absent from task history/logs, and mixed legacy health rows lack explicit normalized/skipped accounting. | Every scheduler attempt that reaches safe storage is observable and health reports bounded compatibility accounting without silently changing denominators. |
| B14 | Closed locally | The published-upgrade gate runs source directly, edits version state manually, and does not execute the generated scheduler command. | A packed 0.9 artifact performs the canonical migration and executes scheduler output against unchanged tasks created by published 0.8.14. |
| B15 | Hosted signoff pending | No native macOS or Windows scheduler acceptance is a required release gate. | Disposable macOS launchd and Windows Task Scheduler CI jobs exercise packaged artifacts with unconditional cleanup; Linux covers the compiled binary outside `PATH`. |

## Review Gates

| Gate | Reviewer focus |
| --- | --- |
| R1 Scheduler contract | Cron-equivalent timing, quoting, runtime context, lifecycle state, and idempotence across cron, launchd, and Task Scheduler. |
| R2 Upgrade contract | Actual 0.8.14 artifacts, canonical migration, source preservation, backups, history, and all task target kinds. |
| R3 Runtime/package contract | Bun, Node, standalone executables, paths with spaces, stripped environments, and native npm launcher behavior. |
| R4 Health/observability | Exact legacy compatibility boundary, persisted failures, metric denominators, and latest-complete snapshot selection. |
| R5 Release gate | Native CI cleanup, artifact identity, non-mocked scheduler execution, and required status wiring. |

## Operational Signoff

These items are not source-code blockers unless a reproducible product defect is found:

- Create and protect the GitHub `release` environment with required reviewers and deployment branch policy; confirm npm trusted publishing or an environment-scoped `NPM_TOKEN`.
- Run the required hosted Linux ARM64, macOS Intel/ARM64, Windows standalone, and Windows packed-Node scheduler jobs.
- Materialize and sync the configured managed sources.
- Repair or remove the unreachable `qwen-9b-shredder` endpoint and the `qwen-9b-splinter` model-ID mismatch.
- Run a post-repair scheduled improve canary only after explicit authorization to restore a live schedule.
