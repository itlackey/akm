# AKM v0.8.0 CLI Revision Spec

## Status

Proposed breaking CLI redesign for `v0.8.0`.

## Release Position

`v0.8.0` is a **hard break** for the AKM CLI UX.

This release intentionally removes the current legacy self-improvement command surface and replaces it with a smaller command model.

There will be:

- no compatibility aliases
- no legacy command shims
- no hidden fallback routing from old commands to new ones
- no retention of legacy self-improvement code paths purely for backward compatibility

If a command is removed in `v0.8.0`, it is removed. Users must adopt the new command tree.

This applies especially to the current fragmented self-improvement surfaces such as:

- `reflect`
- `distill`
- `proposal ...`
- stash-driven `dream` and `evolve` runtime workflows as the canonical path

Low-level implementation primitives may still exist internally if they are the simplest way to implement the new UX, but they are not preserved as public CLI interfaces and are not retained as legacy compatibility surfaces.

## Goals

- collapse the self-improvement experience into a compact CLI UX
- make AKM own memory cleanup and asset improvement directly
- stop relying on stash assets as the canonical runtime path for dream and evolve flows
- preserve the proposal queue as the trust boundary
- keep net-new asset creation explicit and ergonomic
- prune obsolete and redundant derived memories as part of the built-in improvement flow
- consolidate reinforced memory facts into more authoritative knowledge-oriented assets when appropriate
- clarify that knowledge assets are a higher-trust destination than transient memory notes

## Revised Command Tree

```text
akm
├── improve [<type>|<ref>] [--task "..."] [--dry-run] [--target <source>] [--auto-accept safe]
├── propose <type> <name> (--task "..." | --file <path>) [--target <source>]
├── proposals [--status pending|accepted|rejected] [--type <type>] [--ref <ref>]
├── show proposal <id>
├── diff proposal <id> [--target <source>]
├── accept <id> [--target <source>]
└── reject <id> --reason "..."
```

## Command Semantics

### `akm improve`

```text
akm improve
akm improve <type>
akm improve <ref>
```

This is the primary self-improvement command in `v0.8.0`.

Modes:

- `akm improve`
  - improve all eligible assets in scope
  - includes memory cleanup and consolidation as part of the overall improvement pass
  - may promote repeated or reinforced memory facts into more authoritative knowledge proposals
- `akm improve <type>`
  - improve all assets of a given type
  - examples: `memory`, `skill`, `workflow`, `knowledge`
- `akm improve <ref>`
  - improve one specific asset
  - example: `akm improve skill:code-review`

This command replaces the fragmented public UX previously split across `reflect`, `distill`, and stash-driven dream/evolve orchestration.

For memory-focused runs, `improve` should favor:

- pruning obsolete, redundant, and contradicted derived memories
- consolidating repeated signals into cleaner surviving memory assets
- promoting stable, reinforced facts into knowledge-oriented proposals when they should become more authoritative than memory notes
- avoiding naive age-only deletion that could discard still-useful information

Across both improvement intent and deterministic search ranking, `knowledge`
remains more authoritative than `memory`: when evidence is otherwise
comparable, `knowledge` should rank above `memory`, including inferred
`.derived` memories.

### `akm propose`

```text
akm propose <type> <name> --task "..."
akm propose <type> <name> --file <path>
```

Creates a proposal for a brand-new asset.

Input sources:

- `--task <text>`
  - inline prompt/task text
- `--file <path>`
  - read the task/prompt text from a file

Rules:

- exactly one of `--task` or `--file` is required
- passing both is an error
- the file must be read as UTF-8 text
- the file content is treated exactly like the `--task` text input

This replaces clunkier forms such as `improve --new ...`.

### `akm proposals`

Lists proposal queue entries.

This is the queue inbox command.

### `akm accept`

Accepts a proposal and promotes it into the stash.

The verb is intentionally `accept`, not `apply`, to match proposal review language.

### `akm reject`

Rejects a proposal and records the reason.

## Help Text Drafts

### `akm improve --help`

```text
Usage:
  akm improve
  akm improve <type>
  akm improve <ref>

Description:
  Analyze existing AKM assets and generate improvement proposals.

Modes:
  akm improve
      Improve all eligible assets in the current scope.

  akm improve <type>
      Improve all assets of a given type.
      Example: akm improve memory

  akm improve <ref>
      Improve one specific asset.
      Example: akm improve workflow:release-checklist

What it does:
  - reviews feedback and recent history
  - proposes edits to existing assets
  - distills lessons where useful
  - cleans and consolidates memories
  - writes results to the proposal queue

Options:
  --task <text>        Add extra guidance for this improvement pass
  --dry-run            Show planned actions without generating proposals
  --target <source>    Override the write target for accepted proposals
  --auto-accept safe   Automatically accept low-risk proposals

Examples:
  akm improve
  akm improve memory
  akm improve skill
  akm improve skill:code-review
  akm improve workflow:incident-response --task "reduce duplication"
```

### `akm propose --help`

```text
Usage:
  akm propose <type> <name> --task "..."
  akm propose <type> <name> --file <path>

Description:
  Create a proposal for a brand-new AKM asset.

Input:
  --task <text>        Inline task or prompt text
  --file <path>        Read task or prompt text from a file

Rules:
  Exactly one of --task or --file is required.

Examples:
  akm propose skill release-auditor --task "review release artifacts before publish"
  akm propose workflow hotfix-triage --file ./prompts/hotfix-triage.md
```

```text

Examples:
```

## UX Rules

- `improve` is for improving existing content, whether one ref, one type, or the whole stash
- `propose` is for net-new asset creation
- `proposals` is the proposal inbox
- `accept` and `reject` resolve proposals
- `show proposal <id>` and `diff proposal <id>` remain inspection tools

## Dream And Evolve Position

In `v0.8.0`, dream and evolve stop being canonical stash-asset runtime workflows.

Instead:

- dream behavior is absorbed into `akm improve`
- evolve behavior is absorbed into `akm improve`, `akm propose`, `akm proposals`, `akm accept`, and `akm reject`

The built-in flow may still be inspired by auto-dream style mechanics, but `v0.8.0` does not preserve `akm dream` as the public command shape.

Stash assets may still exist as references, documentation, or implementation inspiration, but they are no longer the primary public runtime path.

## Internal Mapping

- `improve <ref>` -> current reflect-like behavior plus optional lesson distillation
- `improve <type>` -> batch improvement over all refs of that type plus type-specific cleanup logic
- `improve` -> stash-wide maintenance pass, including dream-style memory consolidation
- `propose` -> current propose-like behavior with either inline or file-backed task input
- `proposals` -> proposal list behavior
- `accept` -> proposal acceptance behavior
- `reject` -> proposal rejection behavior

These mappings describe behavior, not compatibility promises.

## Breaking Change Notes

### Removed Public Commands

The following public command surfaces are removed from `v0.8.0`:

- `reflect`
- `distill`
- `proposal list`
- `proposal show`
- `proposal diff`
- `proposal accept`
- `proposal reject`

Any additional legacy dream/evolve workflow entrypoints that exist only as stash assets are also no longer considered part of the canonical AKM CLI UX.

### No Compatibility Layer

`v0.8.0` does not provide:

- alias commands
- transitional warnings that still execute old commands
- retained legacy routing just to support old automation

Existing scripts, docs, prompts, and habits must be updated to the new commands.

## Migration Notes

### Command Translation

Old to new:

- `akm reflect <ref>` -> `akm improve <ref>`
- `akm distill <ref>` -> `akm improve <ref>`
- stash-driven dream flow -> `akm improve` or `akm improve memory`
- `akm proposal list` -> `akm proposals`
- `akm proposal show <id>` -> `akm show proposal <id>`
- `akm proposal diff <id>` -> `akm diff proposal <id>`
- `akm proposal accept <id>` -> `akm accept <id>`
- `akm proposal reject <id> --reason "..."` -> `akm reject <id> --reason "..."`

### Documentation Migration

Update all:

- README examples
- CLI docs
- blog posts and walkthroughs
- AGENTS instructions
- stash workflows and command assets that teach the old model
- any automation scripts that shell out to removed commands

### Automation Migration

Any automation that currently calls removed commands must be rewritten before adopting `v0.8.0`.

Examples:

- `reflect` pipelines must call `improve <ref>`
- `proposal accept` pipelines must call `accept`
- prompt files used for net-new proposals should migrate to `akm propose ... --file <path>` when inline strings are awkward or too large

## Recommended Examples

```sh
akm improve
akm improve memory
akm improve skill
akm improve skill:code-review

akm propose workflow incident-triage --task "guide rollback decisions"
akm propose workflow incident-triage --file ./prompts/incident-triage.md

akm proposals
akm show proposal proposal_123
akm diff proposal proposal_123
akm accept proposal_123
akm reject proposal_124 --reason "too broad"
```

## Summary

`v0.8.0` replaces the fragmented self-improvement UX with a compact command family:

- `improve`
- `propose`
- `proposals`
- `accept`
- `reject`

This is a deliberate breaking redesign, not a compatibility refactor.
