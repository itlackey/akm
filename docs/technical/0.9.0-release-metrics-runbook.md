# 0.9.0 Release Metrics Runbook

Use this document to restart the `0.9.0` release-readiness analysis without
re-deriving the metric semantics from scratch.

## Scope

This runbook answers two different questions that are easy to conflate:

1. Is the `0.9.0-beta.50+` telemetry repair working?
2. Is `0.9.0` ready for `rc.0` or for stable release?

They are not the same gate.

## Current release stance

Per the ratified release plan in `docs/reviews/akm-meta-review/CONTEXT.md` and
the later release-session updates:

- The `0.9.0` release train is now on `next` as `rc.N` builds.
- `latest` stays on `0.8.14` until the final stable `0.9.0` cut.
- Stable `0.9.0` waits for all of:
  - a clean-week GRR before-baseline,

  - 30 clean days of `improve_cycle_metrics`.

As of this analysis session, the telemetry repair is working, the release train
has already moved onto `rc` builds, and stable `0.9.0` is not yet justified.

## Metric tiers

### Tier 1: beta.50 / beta.52 telemetry validation

These metrics answer: "did the fix land, and are the counters interpretable?"

- `skippedCount` must exist in `improve_runs.metrics_json`.
- `rejectedCount` must be small, not the old ~13k polluted value.
- `acceptedCount / plannedCount` is meaningful only on rows where the repaired
  telemetry is present.
- The `:40` proactive runs are the best short-window operational smoke test.

### Tier 2: release-governing metrics

These metrics answer: "is stable `0.9.0` ready?"

- North star: `UCE` (useful context events/week).
- Governing number: per-lane 30-day `GRR`.
- Minting lanes stay off below `5%` GRR.
- Stable release requires 30 clean days of `improve_cycle_metrics` after the
  post-beta.58 re-baseline sequence.

### Tier 3: operator diagnostics

These come from `akm health` advisories.

- Some warns are explicitly expected during the minting-shutdown re-baseline.
- They should not be treated as automatic stable-release blockers unless the
  advisory doc says they require action.

## Guardrails and semantic traps

- Do not compare pre-beta.50 `rejectedCount` to post-beta.50 `rejectedCount`.
  Old rows were polluted by gated skips.
- Use `skippedCount IS NOT NULL` as the practical discriminator for repaired
  rows.
- `accepted / planned` is the meaningful run-quality ratio post-fix.
- High `skippedCount` is expected; it is not itself a failure signal.
- `akm health --detail per-run --format md` is invalid. Use `--detail full`
  with JSON and read `windows[]` / `deltas`, or query SQLite directly.
- `akm health` does not surface the full release-governing GRR/UCE picture by
  itself. It is necessary, not sufficient.
- Older chat/session notes may say "stable is unblocked after Group B" or "ship
  stable whenever you want." Treat that as obsolete. The canonical plan is the
  `rc.N` observation/tuning sequence ending only after the clean-week baseline,
  shutdown batch, and 30 clean days.

## Reproduction runbook

### 1. Generate the canonical health artifacts

```sh
bun run src/cli.ts health --since=24h --format json > /tmp/health_24h.json
bun run src/cli.ts health --since=24h --window-compare=24h --detail full --format json > /tmp/health_24h_cmp.json
bun run src/cli.ts health --since=14d --window-compare=14d --detail full --format json > /tmp/health_14d_cmp.json
npm view akm-cli dist-tags --json > /tmp/akm_dist_tags.json
akm --version
```

Read first:

- `status`
- `hardChecks`
- `advisories`
- `improve`
- `windows[]`
- `deltas`
- current `next` tag and local CLI version

### 2. Verify repaired telemetry semantics

```sh
sqlite3 -json "$HOME/.local/share/akm/state.db" "
SELECT id, started_at,
  json_extract(metrics_json,'$.plannedCount') AS plannedCount,
  json_extract(metrics_json,'$.acceptedCount') AS acceptedCount,
  json_extract(metrics_json,'$.rejectedCount') AS rejectedCount,
  json_extract(metrics_json,'$.skippedCount') AS skippedCount,
  json_extract(metrics_json,'$.autoAcceptedCount') AS autoAcceptedCount
FROM improve_runs
ORDER BY started_at DESC
LIMIT 20;"
```

Pass condition for the repair:

- `skippedCount` exists,
- `rejectedCount` is small,
- giant gated volume is in `skippedCount`, not `rejectedCount`.

### 3. Compute the short-window aggregates

```sh
sqlite3 -json "$HOME/.local/share/akm/state.db" "
SELECT COUNT(*) AS runs,
  SUM(CAST(json_extract(metrics_json,'$.plannedCount') AS INTEGER)) AS planned,
  SUM(CAST(json_extract(metrics_json,'$.acceptedCount') AS INTEGER)) AS accepted,
  SUM(CAST(json_extract(metrics_json,'$.rejectedCount') AS INTEGER)) AS rejected,
  SUM(CAST(json_extract(metrics_json,'$.skippedCount') AS INTEGER)) AS skipped,
  SUM(CAST(json_extract(metrics_json,'$.autoAcceptedCount') AS INTEGER)) AS autoAccepted,
  SUM(CAST(json_extract(metrics_json,'$.errorCount') AS INTEGER)) AS errors
FROM improve_runs
WHERE started_at >= datetime('now','-24 hours');"
```

Repeat for `-7 days` and, when needed, `-14 days`.

### 4. Check the `:40` proactive-lane smoke test

```sh
sqlite3 -json "$HOME/.local/share/akm/state.db" "
SELECT started_at,
  json_extract(metrics_json,'$.plannedCount') AS planned,
  json_extract(metrics_json,'$.acceptedCount') AS accepted,
  json_extract(metrics_json,'$.rejectedCount') AS rejected,
  json_extract(metrics_json,'$.skippedCount') AS skipped,
  json_extract(metrics_json,'$.autoAcceptedCount') AS autoAccepted
FROM improve_runs
WHERE dry_run=0
  AND substr(started_at,15,2)='40'
  AND started_at >= '2026-07-05T20:34:00Z'
ORDER BY started_at DESC
LIMIT 30;"
```

Compare against the historical baseline:

- healthy: about `15 planned`, `11-16 accepted`
- current: around `17-18 planned`, `5-6 accepted`

If throughput exists but acceptance is low, the lane is alive but weaker than
its historical healthy state.

### 5. Check cycle-metrics maturity

```sh
sqlite3 -json "$HOME/.local/share/akm/state.db" "
SELECT COUNT(*) AS rows,
  MIN(ts) AS first_ts,
  MAX(ts) AS last_ts,
  AVG(mean_recall) AS avg_recall,
  AVG(distinct_content_ratio) AS avg_distinct_ratio,
  SUM(CASE WHEN json_array_length(alerts_json) > 0 THEN 1 ELSE 0 END) AS alert_rows
FROM improve_cycle_metrics
WHERE ts >= '2026-07-05T20:34:00Z';"
```

Stable-release pass condition is not a single threshold here. The key question
is simply whether the ratified clean observation window is mature. If the answer
is "only a few days of rows," stable is not ready.

### 6. Check proposal queue health

```sh
sqlite3 -json "$HOME/.local/share/akm/state.db" "
SELECT COUNT(*) AS total,
  SUM(CASE WHEN status='pending' THEN 1 ELSE 0 END) AS pending,
  SUM(CASE WHEN status='accepted' THEN 1 ELSE 0 END) AS accepted,
  SUM(CASE WHEN status='rejected' THEN 1 ELSE 0 END) AS rejected
FROM proposals;"
```

This does not decide release readiness by itself, but it tells you whether a bad
health picture is actually causing queue accumulation.

### 7. Finish with the release judgment

Ask these in order:

1. Is repaired telemetry present and interpretable?
2. Do hard health checks pass?
3. Are current warns expected or actionable?
4. Is the `:40` proactive lane alive?
5. Is its acceptance quality close to or far below the historical baseline?
6. Do we have the clean-week GRR baseline yet?
7. Do we have 30 clean days of `improve_cycle_metrics` yet?

If (1) and (2) are yes, the current `rc.N` train is supportable.

If (6) or (7) are no, stable `0.9.0` is not ready regardless of whether the
short-window health snapshot looks acceptable.

fresh data:

- Post-beta.52 telemetry semantics are fixed.
- `accepted / planned` is the meaningful short-window quality ratio.
- Giant `skippedCount` values are expected.
- `outcome-proxy-dead` and `salience-uniformity-collapse` are expected during
  the re-baseline period.
- The release train is already on `rc.N`; check the live dist-tags and local
  version first so you do not reason from stale `rc.0` assumptions.
- Host-hygiene advisories and invalid workflow specs may show up during health /
  reindex work; record them, but do not confuse them with the release-metrics
