## Issues

### [1] `akm info` does not expose `stashDir` / `configPath`
- Section: §1.4
- Command: `akm info --format json | jq '.stashDir, .configPath'`
- Expected: both paths under `$AKM_SANDBOX`
- Actual: both fields are `null` (not present in output). Top-level keys are `assetTypes, indexStats, registries, schemaVersion, searchModes, semanticSearch, sourceProviders, version`. `stashDir` IS returned by `akm init`, just not by `akm info`.
- Severity: minor (checklist drift) — either add the fields to `info` output or update the checklist.
- Notes: spec §9 lists `info` as part of locked CLI surface but does not lock its output shape. Still, the user-facing checklist should match reality.

### [13] All errors exit `78` (CONFIG) regardless of class
- Section: §5, §16, contract per CLAUDE.md "Exit codes"
- Probes:
  - `akm show skill:does-not-exist` → exit 78 (expected GENERAL=1)
  - `akm show foo` → exit 78 (expected USAGE=2)
  - `akm config set sources weird-thing` → exit 78 (probably USAGE=2 or CONFIG=78)
  - `akm remove extra` (missing) → exit 78 (expected GENERAL=1)
  - Writable-on-npm config rejection → exit 78 ✓ (this one is correct)
- Severity: major — exit-code taxonomy is collapsed onto a single value. Per spec, USAGE=2 / CONFIG=78 / GENERAL=1; only true config errors should be 78. Agents using exit codes to differentiate classes (e.g. retry vs fail vs prompt) cannot.

### [14] `akm search` with no query returns hits instead of erroring

- Expected: `UsageError`; help mentions the missing arg.
- Actual: returns `hits[]` with the full fixture set. Treats no-arg as a wildcard.
- Severity: major — inconsistent with the §16 contract; this command must hard-fail.

the issues are doc drift, not product bugs. The doc was written against an
older CLI surface (`config show`, `vault add`, `workflow run --dry-run`,
`registry add-kit`, `clone --to`) and never re-verified before the v0.6.0
cut. Recommend rewriting the checklist against actual CLI output as a
follow-up to this run.

The **product bugs that block release** are the five blockers. Each is a
small, surgical fix:
- (10/17) Map persisted/exposed `kind` literal to the spec values in `src/sources/*` provider classes.
- (11/23) Apply the documented default in the same place the list renderer reads writability.
- (8) Wire `error.hint()` into the JSON error renderer in `src/output/`.
- (21) Add `defaultWriteTarget` cases to `src/commands/config-cli.ts` (parse/get/set/unset).
