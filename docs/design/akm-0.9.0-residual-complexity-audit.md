# Residual Complexity Report — What Survives the 0.9.0 Clean-Up

> **Status (2026-07-13):** historical audit record. Its findings are integrated into the plan (§13), with these dispositions decided after it was written: **§4's scope-downs of bindings, the memory lifecycle, and adapter facets were OVERRULED** by the maintainer reconciliation (DEV-3/4/5 — those ship in 0.9.0, budgeted in the plan's §12.1 ledger); finding #8 (scheduler backends) was **withdrawn** (plan §14.1); finding #3 resolved as CDN-flip with the HTML report kept; the **canary probe/store is carved out** of finding #6's prove-or-delete tier (plan §13.2 — it is the memory-lifecycle gate's harness; only the advisory alert loop remains in the tier); and the synthesize.json "0% accept" note in finding #4 refers to procedural compilation (#615), not recombine (the recombine deletion stands on its own default-off evidence). Read the plan for current scope; read this for the audit reasoning.

## 1. Verdict

Even after the 0.9.0 plan removes the asset taxonomy, wiki subsystem, improve god-functions, and the unproven improve lanes, a **large residual layer of gold-plating survives — roughly 4,000 LOC of confidently-deletable code plus 8,000–10,000 LOC of prove-or-delete subsystems and a vendored 1 MB asset.** It clusters in three places: (a) a **learning/ranking apparatus** that shapes retrieval with a signal the code's own tripwire measures at `corr = +0.0104` (`outcome-loop.ts:56-57`); (b) **default-on heavy subsystems whose value is unmeasured** — graph-extraction (~4288 LOC), the collapse-detector/canary monitor (~900 LOC), and the HTML health report + vendored echarts (~1800 LOC + 1 MB); and (c) **redundant runtime breadth** the plan keeps wholesale — two workflow codecs, ten agent harnesses, three schedulers, a plugin-loader for one fetcher. Separately, the plan's **own new machinery** (bindings/activation, adapter facets, a second supersession representation) is framework-before-second-consumer and should be scoped down before it ships.

## 3. The Big Three Clusters

**A. The learning/ranking apparatus built on a signal measured near zero (~600 LOC + tables).** Findings 7, 9, 13, 15, 17, 18, 20 are one organism: an outcome loop writes `asset_outcome` and `review_pressure` every cycle, an encoding-salience NLP model and a scoped-utility EMA feed sub-scores, and a per-query ablation harness exists to measure their contribution — all to shape `rankScore`. But the code's own tripwire reports `corr(outcome_score, accepted_change_rate) = +0.0104` at n=5,706 and emits `outcome_proxy_dead` calling it "noise, not signal" (`preparation.ts:1678-1690`), while the ablation harness admits it reads Δ=0 under the saturation regime the store actually runs in (`ranking-contributors.ts:518-522`). **Prove-or-delete as a unit:** set `w_o=0`, run the nDCG/saturation harness once; if outcome/encoding/scoped signals don't move rankings, delete the outcome loop, `review_pressure`, the parity triple + flag, the scoped table, and the ablation branch, and fall encoding back to the existing `DEFAULT_TYPE_ENCODING_WEIGHTS` stub. This removes machinery whose value is not merely unproven but measured absent.

**B. Default-on heavy subsystems whose payoff is never measured (~7,000 LOC + 1 MB).** Graph-extraction (4288 LOC, per-batch LLM cost, one conditional ranking boost with no nDCG proof — finding 1), the collapse-detector/canary monitor (~900 LOC running FTS probes + full scans every cycle to emit advisory alerts that block nothing — finding 6), and the HTML health report with a vendored 1 MB `echarts.min.js` inlined into every report to render one command (finding 3). Each is default-on and each guards or serves a workload the same evidence base shows is barely happening (`proactive-maintenance-issue.md`: ~0 actions/run, 95.9% assets untouched). **Prove-or-delete each against the curate-golden harness:** graph boost must show a rank delta or go default-off; the collapse detector must have caught one real event or reduce to a single cheap metric; the HTML report should be deleted in favor of the JSON + md paths that already exist.

**C. Redundant runtime breadth kept wholesale (~3,700 LOC).** Two workflow codecs compiling to the identical IR where markdown is a strict subset of YAML (~650, finding 5); ten agent harnesses of which seven were appended as a "capability matrix" with no product consumer beyond their own unit tests (~2000, finding 2); two OpenCode harnesses for one tool (~200, finding 10); three OS scheduler backends where the embedded runner covers the common case (~500, finding 8); a filesystem plugin-loader for a one-element fetcher registry (~55, finding 21). **Collapse to the proven path in each pair/set:** one codec (YAML), the three harnesses with real users, one OpenCode mode, cron+embedded, and an inlined YouTube fetcher.

## 4. Watch the Plan's Own Additions

The plan is not only preserving old gold-plating — it introduces new machinery that is itself framework-before-second-consumer. Be skeptical of these before they ship:

- **Bindings/activation reframe (install→bind→enable, `workspace_bindings`, export digests, trust decisions).** Grep for `workspace_bindings`/`bindWorkspace` returns **zero** hits today; the only "binding" in code is unrelated env-var binding (`env/env-binding.ts`). The plan mints a three-stage lifecycle + table + trust machinery for a system that today activates implicitly with essentially one workspace consumer. **Scoped alternative:** keep implicit activation; do not add `workspace_bindings`/export-digest/trust until a real multi-workspace consumer exists. (prevents ~150–300 new LOC)

- **Adapter facets (`BundleAdapter` + `MemoryLifecycleAdapter`/`AuthoringAdapter`/`ExportAdapter`).** None exist in `src` yet. The plan replaces one static registry (`asset-registry.ts` TYPE_TO_RENDERER) with a 6-method facet fanned across 14 formats — and concedes 6 of 14 supply pure constant renderer/action maps; a memory-lifecycle facet would have exactly one implementer. **Scoped alternative:** keep a single data-driven format table for the trivial facets; write per-format code only where `recognize`/`validateL1` genuinely differ. Do not mint a one-implementer lifecycle facet. (prevents ~300–500 new LOC)

- **A second supersession representation (§S6 in-place `supersededBy` + rank demotion).** Supersession is already encoded once by `archiveMemory` (`consolidate.ts:838`): archive-dir move + `superseded_by` frontmatter + git history, already called by merge/delete/promote ops. Adding an in-place `supersededBy`+demotion path gives two non-destructive encodings of the same fact that every downstream reader must understand. **Scoped alternative:** reuse `archiveMemory` through the new FileChange transaction; do not add a parallel encoding. (prevents ~80–150 duplicate LOC)

- **Keeping the outcome-weight lane behind a "reversible parity flag" (§9.3).** The plan explicitly defers the flip and keeps the table + update loop + CASE guards "untouched" for a +0.15 term it admits is "LIVE-but-unproven." Per cluster A above, this is machinery preserved on hope; the flag has no `.default()` and is read via triple-negative `!== false` in three files. Run the harness now rather than deferring.

## 5. Leave It Alone

- **Embeddings + hybrid ranking core.** The base embedder backends and the FTS+vector ranking path have real, broadly-used consumers; only the *env-gated deterministic backend* (finding 14) and the *dead outcome/scoped contributors* are cuttable. Do not touch the primary retrieval path.
- **The three OS scheduler backends are borderline, not clearly wrong.** Each of cron/launchd/schtasks is genuinely exercised on its own platform (`backends/index.ts:73-82`). This is real cross-platform need, not speculative generality — the recommendation (finding 8) is low-confidence and conditional on the embedded runner truly covering the case. Do not cut launchd/schtasks without confirming that.
- **`human`/`agent` shape axis and the JSON output envelope.** Only the `summary` third value (finding 23) and the `md` alias (finding 26) are degenerate; the two-value projection and JSON shaping are load-bearing across commands.
- **`archiveMemory` and the extract/consolidate core.** These are the ~4 processes with proven live output; the config-breadth finding (16) trims around them, not through them.
- **The eval-metrics math itself (`ndcg`/`recall`/`mrr`).** The functions are fine and needed by the bench — the finding (11) is only that they live in the product bundle; relocate, don't delete.

## 6. Total Removable & Sequencing

**Confidently deletable now (high-confidence findings 3, 4, 5, 7, 11, 14, 17, 21, 25 + the collapse of 18):** ~4,300 LOC plus the 1 MB `echarts.min.js` asset. **Prove-or-delete tier (findings 1, 2, 6, 8, 9, 10, 13, 15, 16, 20 — lower confidence, needs one harness run or usage confirmation):** a further ~6,000–8,000 LOC. **Total addressable beyond the plan: roughly 4,300 LOC removable immediately and up to ~12,000 LOC if the prove-or-delete subsystems fail their harness checks**, on top of what 0.9.0 already deletes.

**Sequencing:** Fold the *confident deletions* into the plan's existing chunks now — they are pure removals with no design decision (dead API, env-gated backend, one-element registry, duplicate codec/strategy, the vendored megabyte). Run the *prove-or-delete tier as a single follow-up "0.9.1 measurement pass"** gated on one nDCG/MRR + saturation harness run against the curate-golden set: that one measurement resolves cluster A (outcome/encoding/scoped), the graph boost, and the collapse detector's value at once, so batch them rather than litigating each in the plan. Finally, **hold the plan's new bindings/facets/supersession machinery to the scoped-down alternatives in §4 before it lands** — cheaper to not build than to remove later.
